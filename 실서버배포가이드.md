## 1. Ubuntu Server에 Docker / Docker Compose 설치

> Ubuntu 20.04 / 22.04 / 24.04 LTS 공통. Docker 공식 apt 저장소를 사용한다 (배포판 기본 `docker.io` 패키지는 버전이 오래되어 compose v2 호환성 이슈가 있을 수 있음).

### 1-1. 기존 구버전 제거 (안전)

```bash
sudo apt-get remove -y docker docker-engine docker.io containerd runc || true
```

이미 설치된 적이 없다면 그냥 통과한다.

### 1-2. 의존 패키지 설치

```bash
sudo apt-get update
sudo apt-get install -y ca-certificates curl gnupg lsb-release
```

### 1-3. Docker 공식 GPG 키 + 저장소 등록

```bash
sudo install -m 0755 -d /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg \
  | sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg
sudo chmod a+r /etc/apt/keyrings/docker.gpg

echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] \
  https://download.docker.com/linux/ubuntu \
  $(. /etc/os-release && echo "$VERSION_CODENAME") stable" \
  | sudo tee /etc/apt/sources.list.d/docker.list > /dev/null
```

### 1-4. Docker Engine + Compose 플러그인 설치

```bash
sudo apt-get update
sudo apt-get install -y \
  docker-ce docker-ce-cli containerd.io \
  docker-buildx-plugin docker-compose-plugin
```

`docker-compose-plugin`이 `docker compose` (공백) v2 형태로 제공된다. 본 프로젝트는 모두 `docker compose` (v2) 명령으로 작성되어 있으므로 구식 `docker-compose` (하이픈) 패키지는 설치하지 않는다.

### 1-5. 비root 사용자에게 docker 그룹 부여 (선택, 권장)

```bash
sudo usermod -aG docker $USER
# 로그아웃 후 재로그인하거나
newgrp docker
```

이후 `sudo` 없이 `docker ...` 실행 가능. 아니면 모든 명령에 `sudo`를 붙이면 된다.

### 1-6. 설치 검증

```bash
docker --version
docker compose version
docker run --rm hello-world
```

- `Docker version 27.x` 이상, `Docker Compose version v2.x` 이상이면 OK.
- `hello-world` 가 정상 출력되면 데몬·네트워크·이미지 풀 모두 정상.

### 1-7. 방화벽(UFW) 설정 — 80 포트 개방

UFW가 활성화돼 있는지 먼저 확인:

```bash
sudo ufw status
```

`inactive` 면 그대로 두고 1-8로 진행. `active` 면 다음을 추가:

```bash
sudo ufw allow OpenSSH       # SSH 22 (이미 열려있을 가능성 높음)
sudo ufw allow 80/tcp        # 학생 폰 접근용 — 가이드의 핵심
sudo ufw reload
sudo ufw status verbose
```

`80/tcp ALLOW Anywhere` 가 보이면 OK.

### 1-8. 도커 데몬 부팅 자동 시작 보장

```bash
sudo systemctl enable docker
sudo systemctl status docker
```

`active (running)` 확인. (대부분 1-4 단계에서 자동 enable 됨.)

## 3. 환경 변수 설정 (`.env`)

`.env`는 절대 깃에 안 들어가므로 서버에서 직접 만든다.

### 3-1. `.env.example` 복사

```bash
cp .env.example .env
```

### 3-2. `SESSION_SECRET` 생성 (필수)

세션 쿠키 서명용 32바이트 hex. 다음 한 줄로 생성:

```bash
docker run --rm node:20-alpine node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"
```

출력된 64자리 hex 문자열을 복사.

### 3-3. `.env` 편집

```bash
nano .env
```

다음 항목을 검토·수정:

```ini
NODE_ENV=production
PORT=3000

# ↓ 방금 생성한 hex 64자 붙여넣기
SESSION_SECRET=<여기에 hex 64자>

# 미지정으로 두면 첫 부팅 시 stdout에 한 번 출력됨 (권장).
# 직접 지정하려면 6자리 숫자.
DEFAULT_ADMIN_PIN=

# 추가 관리자(선택). scripts/admin-add.js 로 생성한 scrypt 해시.
ADMIN_PINS=[]

# 자동 ZIP 스냅샷 주기 (분). 기본 120.
AUTO_SNAPSHOT_INTERVAL_MIN=120

# named volume 내부 경로 — 그대로 둘 것.
DB_PATH=/data/order.sqlite
DATA_DIR=/data

LOG_LEVEL=info
```

`docker-compose.yml`에서 추가로 강제하는 값들 (override됨, 신경 안 써도 됨):

- `NODE_ENV=production` — `.env`에 무엇이 있든 운영은 production
- `PORT=3000`
- `DB_PATH=/data/order.sqlite`
- `BACKUP_DIR=/data/backups` — 자동 ZIP 백업 위치 (named volume 안)
- `SESSION_COOKIE_SECURE=false` — **HTTP 운영 기본값.** HTTPS 도입 시 §8 참조.

`.env` 권한 잠그기:

```bash
chmod 600 .env
```

또는


## 5. ★ ADR-033 사이드체크 (반드시 수행)

**2026-05-17 사고 재발 방지.** CLOSED 가드가 정적 자산을 가로채던 회귀가 host 테스트만 통과하고 prod에 들어간 적이 있다. 운영 컨테이너에 직접 HTTP를 쏴서 정적 자산이 200으로 떠야 OK.

```bash
curl -sI http://localhost/web-logo.png
curl -sI http://localhost/mascot/mascot.png
curl -sI http://localhost/items/adrenaline.webp
```

세 응답 모두 다음 두 조건 만족해야 함:

- `HTTP/1.1 200 OK`
- `Content-Type: image/*`

`302` / `423` / `text/html` 이 보이면 → `server/middleware/business-state.js`의 `STATIC_ASSET_EXT` 화이트리스트 누락 회귀. ADR-033 회귀 테스트(`server/middleware/__tests__/business-state.test.js`)도 같이 확인.

## 8. HTTPS 도입(선택) — 학교 도메인 + 인증서 있는 경우

부스 운영 기본 가정은 **HTTP**다 (`SESSION_COOKIE_SECURE=false`, ADR-031). 학교에서 도메인 + 인증서를 제공하는 경우에만 다음을 추가.

옵션 1 — **상위 reverse proxy (학교 nginx, Cloudflare 등)에서 TLS 종단**

이 경우 학교측 nginx가 HTTPS를 받고 `http://211.171.114.154:80` 으로 평문 전달. 이 모드라면:

```bash
# .env 에서
SESSION_COOKIE_SECURE=true
```

그리고 `server/app.js`의 `app.set('trust proxy', 1)` 한 줄이 활성화돼 있어야 함 (이미 P0-A에서 추가된 흔적이 있으면 OK; 없으면 동일 한 줄 추가 후 빌드).

옵션 2 — **이 서버 자체에서 TLS 종단** (Let's Encrypt)

운영 일정상 권장하지 않음. 굳이 한다면 `caddy` 컨테이너를 nginx 앞에 추가하거나, nginx 컨테이너 conf에 certbot volume을 마운트. 부스 운영 일정(이틀)에는 oversized. 학교 도메인 reverse proxy가 안 되면 그냥 HTTP로 운영하는 게 안전하다.

## 9. 백업 / 복구

### 9-1. 자동 백업 — 별도 조치 불요

`AUTO_SNAPSHOT_INTERVAL_MIN=120` (기본 2시간) 마다 named volume 안쪽 `/data/backups/` 에 ZIP 6개 회전 저장. 컨테이너 재생성·이미지 재빌드 후에도 보존됨 (P1-2, Codex v3).

확인:

```bash
docker compose exec app ls -la /data/backups
```

### 9-2. 수동 ZIP 다운로드 (정산 후)

`http://211.171.114.154/admin/settlement` → "📦 ZIP 다운로드" 버튼 → 로컬 PC로 저장 → 학생회 클라우드 업로드.

### 9-3. 호스트로 named volume 통째 백업

```bash
# 1일차 종료 직후 등 중요한 시점에 1회
sudo docker run --rm \
  -v chickenedak-data:/data \
  -v "$PWD":/backup \
  alpine tar czf /backup/chickenedak-data-$(date +%Y%m%d-%H%M).tar.gz -C /data .
```

생성된 `.tar.gz` 를 학생회 클라우드/USB 등 외부 위치에 추가 보관.

### 9-4. 복구

```bash
docker compose down
sudo docker run --rm \
  -v chickenedak-data:/data \
  -v "$PWD":/backup \
  alpine sh -c "cd /data && tar xzf /backup/chickenedak-data-<TIMESTAMP>.tar.gz"
docker compose up -d
```

## 10. 트러블슈팅

| 증상 | 원인 가능성 / 대응 |
| --- | --- |
| `docker compose up -d --build` 가 빌드 단계에서 실패 | 메모리 부족(특히 t2.micro급) → swap 추가하거나 더 큰 인스턴스. better-sqlite3 빌드 도구는 Dockerfile에 이미 포함됨. |
| `docker compose ps` 가 `unhealthy` | `docker compose logs app` 확인. 대부분 `.env`의 `SESSION_SECRET` 누락 또는 `DB_PATH` 쓰기 권한 문제. |
| 외부 PC에서 `http://211.171.114.154/` 가 timeout | (a) 클라우드 콘솔 보안그룹 80 포트, (b) 서버 UFW, (c) 컨테이너 상태 — 이 순서로 확인. |
| `curl http://localhost/healthz` 만 OK, 외부 X | UFW 또는 상위 방화벽이 80을 막음. `sudo ufw status` + 클라우드 콘솔. |
| 정적 자산 200 OK가 아닌 302/423 | ADR-033 회귀. `server/middleware/business-state.js` `STATIC_ASSET_EXT` 확장자 누락 가능. dev 컨테이너에서 `npm test` 재실행해 회귀 위치 특정. |
| PIN 분실 | `docker compose exec app sh` → `sqlite3 /data/order.sqlite "DELETE FROM admins;"` → `docker compose restart app` → 로그에서 새 PIN 메모. |
| 시간대가 UTC로 보임 | 컨테이너는 UTC 기본. `operating_date` 등 비즈니스 로직은 KST 기준으로 처리하도록 코드에 내장돼 있으므로 보통 무시 가능. 로그 시각을 KST로 보고 싶으면 `docker-compose.yml`의 `app.environment`에 `TZ: Asia/Seoul` 추가 후 `docker compose up -d` 재기동. |
| 한글 깨짐 | Ubuntu locale 미설정. `sudo apt-get install -y locales && sudo locale-gen ko_KR.UTF-8` 후 SSH 재접속. |
