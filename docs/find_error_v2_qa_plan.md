# find_error_v2 검수 지시서

## 검수 목적

이번 v2 검수의 목적은 실제 축제 현장에서 사용자와 운영진의 혼란을 줄이는지 확인하는 것이다.

특히 다음 흐름을 중점 확인한다.

- 주문은 9자리 학번 기준으로 정상 접수되는가
- 쿠폰 대상 검증은 주문 가능 여부와 분리되는가
- 이체 완료 요청 중복 제출에서 내부 에러가 사용자에게 보이지 않는가
- 본부 대시보드에서 운영자가 주문 항목과 상태를 빠르게 확인할 수 있는가
- 내역/쿠폰 탭이 운영 확인에 필요한 정보를 제공하는가
- 주문이 없는 상태에서도 관리자 화면 구조가 유지되는가

## 검수 시나리오

### 시나리오 1. 추천 UI 제거

- 학생회 추천 BEST 영역 아래에 작게 줍기 가능한 추천 메뉴가 더 이상 보이지 않는지 확인
- 추천 BEST 배너 자체는 깨지지 않고 표시되는지 확인
- 일반 메뉴 카드에서는 여전히 `줍기`가 가능한지 확인
- `추천` 카테고리 탭을 유지했다면 해당 탭에서 추천 메뉴가 정상 표시되는지 확인

### 시나리오 2. 관리자 내역 탭

- 어드민 페이지에 내역 탭이 보이는지
- 주문 상태 변경 로그가 보이는지
- 시간/주문번호/상태 변경/처리 주체가 보이는지
- 주문 상태를 변경하면 내역에 기록되는지
- 새로고침 후에도 내역이 유지되는지
- 내역이 없는 경우 빈 상태가 자연스럽게 표시되는지
- 관리자 인증이 풀린 상태에서 내역 API 접근이 막히는지 확인

### 시나리오 3. 관리자 쿠폰 탭

- 어드민 페이지에 쿠폰 탭이 보이는지
- 쿠폰 사용 시간이 보이는지
- 누가 어떤 주문에서 쿠폰을 사용했는지 보이는지
- 쿠폰 미사용 주문은 쿠폰 내역에 잘못 표시되지 않는지
- 쿠폰명과 할인 금액이 보이는지
- 쿠폰 사용 건수와 정산 탭의 쿠폰 건수가 일치하는지
- 쿠폰 내역이 없는 경우 빈 상태가 자연스럽게 표시되는지

### 시나리오 4. 학번 검증

- 9자리 숫자 학번은 주문 가능
- `202111123` 같은 `37`이 없는 학번도 주문 가능
- 숫자 8자리/10자리/문자 포함 학번은 주문 불가
- 쿠폰은 `****37***` 패턴일 때만 사용 가능
- 쿠폰 대상이 아니어도 일반 주문은 가능
- 쿠폰 대상이 아닌 경우 주문 자체가 막히지 않는지 확인
- 쿠폰 대상이 아닌 학번에서 쿠폰 할인 금액이 적용되지 않는지 확인
- 백엔드 직접 요청에서도 쿠폰 미사용 + non-37 9자리 학번 주문이 성공하는지 확인

### 시나리오 5. 주문 상태 화면

- 접수/입금/확인/조리/수령 시간 리스트가 보이지 않는지
- ISO timestamp가 사용자 화면에 노출되지 않는지
- 닭고기 이모지, 체크 이모지가 보이지 않는지
- 주문 상태 핵심 정보는 여전히 확인 가능한지
- `ORDERED`, `TRANSFER_REPORTED`, `PAID`, `COOKING`, `READY`, `DONE`, `HOLD`, `CANCELED` 상태별 핵심 문구가 깨지지 않는지
- 모바일 화면에서 ready banner와 sticky 상태 영역이 겹치지 않는지

### 시나리오 6. 이체 완료 요청 화면

- 불필요한 안내 문구가 제거되었는지
- `ORDERED` 주문은 이체 완료 요청 가능
- `TRANSFER_REPORTED` 주문에서 다시 제출해도 “불법 상태 전이”가 보이지 않는지
- 대신 친절한 안내 문구가 보이는지
- 사용자 문구가 “이미 이체 완료 요청이 접수됐어요. 본부 확인을 기다려주세요.”와 같은 의미인지 확인
- 중복 제출 후 DB 상태가 변경되지 않는지 확인
- `PAID/COOKING/DONE/CANCELED` 상태는 다시 이체 요청 불가
- `HOLD` 상태도 v2 정책대로 사용자 재요청이 불가한지 확인

### 시나리오 7. 관리자 대시보드 카드

- 주문번호/가격/이름/상태/경과 시간/주문 항목이 모두 보이는지
- 주문 항목이 여러 개일 때 UI가 깨지지 않는지
- 주문 항목이 3개 이상일 때 축약 표시가 자연스러운지
- 경과 시간이 -1분으로 시작하지 않고 0분부터 시작하는지
- HOLD 상태 버튼이 “재확인”이 아니라 “이체 확인”으로 표시되는지
- 카드 클릭 시 상세 페이지로 이동하지 않는지
- 상태 변경 버튼은 정상 동작하는지
- 액션 버튼 클릭이 카드 클릭 이벤트와 충돌하지 않는지
- 카드가 클릭 가능한 것처럼 pointer cursor로 보이지 않는지

### 시나리오 8. 장사 시작 UI

- 장사 시작(시간 전) 버튼이 과도하게 긴 UI로 보이지 않는지
- 시간 전 상태의 안내 문구와 버튼이 모바일에서 겹치지 않는지
- 장사 시작 버튼 클릭 시 기존처럼 영업 상태가 OPEN으로 바뀌는지
- 장사 시작 후 주문이 없어도 6개 상태 박스가 보이는지
- 각 박스의 빈 상태가 자연스럽게 표시되는지
- 주문이 들어오면 해당 상태 박스에 카드가 정상 표시되는지

## 자동 테스트 제안

가능하면 다음 테스트를 추가하거나 기존 테스트를 갱신한다.

- 학번 9자리 검증 테스트
- 쿠폰 대상 학번 패턴 테스트
- 쿠폰 미대상 학번도 일반 주문 가능 테스트
- 쿠폰 미대상 학번에서 쿠폰 할인 미적용 테스트
- 백엔드 `POST /api/orders`의 9자리 숫자 검증 테스트
- 백엔드 쿠폰 사용 시 `****37***` 패턴 검증 테스트
- 이체 완료 요청 중복 제출 UX 테스트
- 불법 상태 전이 내부 메시지 미노출 테스트
- `TRANSFER_REPORTED` 중복 제출 시 DB 상태/timestamp 미변경 테스트
- `PAID/COOKING/READY/DONE/CANCELED/HOLD` 재요청 차단 테스트
- 상태 화면 timeline 숨김 테스트
- 사용자 status 화면 ISO timestamp 미노출 테스트
- READY 상태 이모지 제거 테스트
- 이체 완료 요청 안내 문구 제거 테스트
- 관리자 카드 주문 항목 표시 테스트
- 주문 항목 축약 표시 테스트
- `elapsedMinutes` 0분 clamp 테스트
- HOLD 액션 라벨 테스트
- 카드 클릭 상세 이동 방지 테스트
- 상태 변경 버튼 동작 테스트
- 내역 탭 렌더링/API 테스트
- 주문 상태 변경 시 내역 기록 테스트
- 쿠폰 탭 렌더링/API 테스트
- 쿠폰 미사용 주문 미표시 테스트
- 빈 대시보드 6개 컬럼 유지 테스트
- 장사 시작 CTA compact UI 렌더 테스트

## 최종 통과 기준

- 주문이 9자리 학번 기준으로 정상 접수됨
- 쿠폰 대상 검증과 주문 가능 검증이 분리됨
- 이체 완료 요청 중복 제출 시 사용자에게 내부 에러가 보이지 않음
- 관리자 대시보드에서 주문 항목과 상태 변경이 명확함
- 내역/쿠폰 탭이 design_bundle 기준 의도대로 동작함
- 주문이 없는 상태에서도 운영자가 대시보드를 이해할 수 있음
- `test/lint/build` 통과
- 모바일 화면에서 주요 UI가 깨지지 않음
- 실제 소스 수정 후 회귀 테스트에서 기존 주문/정산/관리자 로그인 흐름이 유지됨
