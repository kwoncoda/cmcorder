# Task 0.6 — Docker compose + named volume

## 목표

- IMPLEMENTATION_PLAN.md §0 Task 0.6 완수: `Dockerfile` + `docker-compose.yml` + 운영용 `.env` 예시.
- 완료 기준: `docker compose up -d` → `http://localhost:3000/healthz` 응답, `restart: always`, named volume `chickenedak-data`, `.env`에 `SESSION_SECRET` · `DEFAULT_ADMIN_PIN` · `AUTO_SNAPSHOT_INTERVAL_MIN=120` 키 노출.
- ADR-023 (Docker compose 단일 컨테이너 운영) 준수.

## 만든 것

- `Dockerfile` — 3-stage build (frontend-build → prod-deps → runtime, node:20-alpine).
- `docker-compose.yml` — 단일 `app` 서비스. named volume `chickenedak-data:/data`, healthcheck, `restart: always`.
- `.dockerignore` — node_modules · dist · .env · docs · tests 등 이미지 컨텍스트에서 제외.
- `.env.example` — 운영용 환경 변수 템플릿 (`SESSION_SECRET`, `DEFAULT_ADMIN_PIN`, `AUTO_SNAPSHOT_INTERVAL_MIN=120`, `DB_PATH`, `DATA_DIR`, `LOG_LEVEL`).

## 한 일

### 1. Dockerfile — multi-stage

3단계 구성:

1. **frontend-build (node:20-alpine)**: `package*.json` 먼저 복사 후 `npm ci --include=dev` → 빌드 입력(`index.html`, `vite.config.js`, `tailwind.config.js`, `postcss.config.js`, `src/`, `public/`, `config/`) 만 복사 → `npm run build`. 서버 코드는 이 단계에 포함 X.
2. **prod-deps (node:20-alpine)**: `npm ci --omit=dev`로 production node_modules만 추출.
3. **runtime (node:20-alpine)**: `server/` + `--from=frontend-build /app/dist` + `--from=prod-deps /app/node_modules` 조립. `mkdir -p /data && chown -R node:node /data` 후 `USER node`로 권한 강하. `EXPOSE 3000`, `HEALTHCHECK wget /healthz`, `CMD ["node", "server/server.js"]`.

**결정**:
- Express의 dist/ 정적 서빙 라우트는 이번 Task에서 추가 X — Phase 6.6 사용자 API 합류 시 같이 처리. dist는 *컨테이너 안에 포함*만 해두고 라우트는 후속.
- alpine 베이스의 busybox `wget`이 기본 포함 → healthcheck용 별도 패키지 설치 불필요(curl도 가능하나 wget이 더 가벼움).
- non-root `node` 사용자 — node:20-alpine 기본 제공. `/data` 소유자만 미리 잡고 USER 선언.

### 2. docker-compose.yml

- `image: chickenedak:latest` + `container_name: chickenedak` — `docker compose logs chickenedak` / `docker exec`이 용이.
- `env_file: .env` 로 보안 변수 주입 + `environment`로 `NODE_ENV=production` · `PORT=3000` · `DB_PATH=/data/order.sqlite` 강제 override (로컬 `.env`의 development 잔재 무력화).
- `volumes: chickenedak-data:/data` — SQLite 파일이 컨테이너 재기동/이미지 교체에도 살아남도록 named volume.
- `healthcheck: wget -qO- http://localhost:3000/healthz` — interval 30s, timeout 3s, retries 3, start_period 10s.
- `restart: always` — 부스 운영 중 프로세스 죽으면 자동 복구.

### 3. .dockerignore

핵심: `.env*` (모든 변형), `node_modules/`, `dist/`, `.git/`, `docs/`, `tests/`, `.claude/`, `.agents/`, `comorder*.zip` 제외. 이미지 슬림화 + 비밀 유출 방지. `.env.example`도 *이미지에는* 안 들어가야 함 — compose는 호스트의 `.env`를 `env_file`로 직접 읽기에 이미지 내부에 필요 없음.

### 4. .env.example

기존 `.env`는 dev용(`NODE_ENV=development`, `AUTO_SNAPSHOT_INTERVAL_MIN=30`). 운영 템플릿은 별도 파일로:

- `SESSION_SECRET=___CHANGE_ME___` — 생성 가이드 주석.
- `DEFAULT_ADMIN_PIN=` — ADR-024에 따라 미지정 시 첫 부팅 자동 생성/stdout 출력.
- `AUTO_SNAPSHOT_INTERVAL_MIN=120` — ADR-022 변경 사항 반영.
- `DB_PATH=/data/order.sqlite`, `DATA_DIR=/data` — Docker volume 경로.
- `ADMIN_PINS=[]` — 후속 admin 계정 등록 형식 안내.

## 테스트 결과

### A. docker-compose.yml 문법 검증

`docker compose config` 실행 → 파싱 성공, 정규화 출력 확인:
- `restart: always` ✓
- `volumes: chickenedak-data` named volume ✓ (정규화 명: `09_order_chickenedak-data`)
- healthcheck CMD `wget -qO- http://localhost:3000/healthz` ✓
- env_file의 `.env` 키들이 environment에 머지 ✓
- `NODE_ENV: production`, `PORT: "3000"`, `DB_PATH: /data/order.sqlite`가 .env의 dev 값을 override ✓

### B. Docker 이미지 빌드

`docker compose build` 시도 → Docker Desktop daemon이 미가동(`error during connect ... dockerDesktopLinuxEngine`). Docker CLI 자체는 v27.2.0 / compose v2.29.2 정상 설치.

**미수행 사유 + 위임**: Docker Desktop 미가동으로 실 이미지 빌드 / `up -d` / `/healthz` 200 응답 확인은 보류. **D-1 리허설(Task 7.5)에서 부스 노트북에 Docker Desktop 가동 후 일괄 검증**한다. 그 시점에서 문제가 발견되면 Phase 7 안에서 보정 PR.

위험도 평가: Dockerfile/compose 문법은 정적 검증을 통과했고, 베이스 이미지(node:20-alpine)·alpine wget·multi-stage COPY는 표준 패턴. 빌드 실패 가능성은 낮음.

### C. 클라이언트/서버 회귀 sanity

`npm test` → `Test Files 5 passed (5) / Tests 33 passed (33)`. 이번 변경(Dockerfile · compose · .dockerignore · .env.example)은 런타임 코드 0건 영향이므로 예상대로 회귀 X.
