## 한 일

### 1. CustomerLayout 컴포넌트 (`src/components/layouts/CustomerLayout.jsx`)
- **헤더**: `Link to="/menu"` 로고("🍗 치킨이닭") + `Link to="/map"` 지도 아이콘 (`data-testid="header-map-link"`, `aria-label="부스 약도 열기"`)
- **본문**: `<Outlet/>`으로 자식 라우트 렌더
- **423 가드**: `useGlobalErrorHandler({ onBusinessClosed })` 등록
  - `useNavigate` + `useLocation` 으로 현재 경로 판별
  - 진행 중 주문 페이지(`/orders/<id>/{complete|transfer|status}`) 정규식 매칭 시 redirect 차단
  - 그 외 경로에서 423이 발생하면 `navigate('/closed', { replace: true })`
  - `useCallback`으로 `handleBusinessClosed` 안정화 → useGlobalErrorHandler의 useEffect deps 재등록 최소화

### 2. App.jsx 라우트 갱신
- 사용자 9 라우트(`/`, `/menu`, `/cart`, `/checkout`, `/orders/:id/{complete,transfer,status}`, `/map`, `/closed`)를 `<Route element={<CustomerLayout />}>` 자식으로 이동
- 관리자 6 라우트와 404 catch-all은 *그대로* (Phase 5에서 AdminLayout 별도 도입 예정)
- `CustomerLayout` import 추가

### 3. 폴링 *명시적 미도입*
- IMPLEMENTATION_PLAN §4.1 변경 사유:
  - 60초 폴링 = 200명 동시 접속 시 분당 200req 의미 없는 트래픽
  - Task 6.8 middleware가 모든 POST를 423으로 거부 → 사용자 API 호출 자체가 reactive 신호
  - 메뉴 fetch·이체·주문 *어디서든* 시도하면 자동 감지 → 폴링 불필요
- `setInterval(fetchBusinessState, ...)` 패턴 코드 0개
