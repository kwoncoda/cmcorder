## 5. 서버 동작 검증

### 5.1 이체 신고 API

브라우저 또는 curl로 TransferPage 폼 제출 후 다음 확인:

```bash
# admin DB 직접 조회 (운영 컨테이너 진입)
docker compose exec app sh
# 컨테이너 안에서
sqlite3 /data/orders.db "SELECT id, depositor_name, use_other_name, other_name FROM orders WHERE status='TRANSFER_REPORTED' ORDER BY id DESC LIMIT 3;"
```

**기대치**: 신규 행에 `use_other_name=0`, `other_name=NULL` 저장. 기존 행은 그대로 보존.

### 5.2 매칭 동작

이체 매칭이 depositor_name 단일로만 동작하는지 확인. transfer-matching 호출이 일어나는 admin 측 매칭 시나리오 점검 (운영 카드 `docs/operations/admin-card.md` 참고).

## 6. 회귀 안전성 점검 (수동 grep)

```bash
docker compose -f docker-compose.dev.yml exec dev sh
# 컨테이너 안에서
grep -rn "use_other_name\|useOtherName\|other_name\|otherName" --include="*.js" --include="*.jsx" . | grep -v node_modules | grep -v dist
```

**기대치**:
- 클라이언트(`src/`): 0건 (TransferReportForm.jsx 및 테스트에서 모두 제거)
- 서버 도메인(`server/domain/`): 0건 (transfer-matching.js에서 단일화)
- 서버 라우트/레포: 0건 (제거 완료) — 또는 default 값 INSERT 의도로 명시한 한 줄만 남음
- DB init.sql: 컬럼 정의는 그대로 (의도된 비범위)

**유의**: docs/* 의 과거 작업 로그 (`docs/tasks/2026-05-14-*`)에는 use_other_name 언급이 있을 수 있으나, 이는 history이므로 그대로 보존.

## 8. 롤백 절차

문제 발생 시:
1. `git revert <commit-sha>` 또는 변경 파일 단위로 `git checkout HEAD -- <file>` 복원.
2. `docker compose build app && docker compose up -d`.
3. `docker compose -f docker-compose.dev.yml exec dev npm test` 회귀 통과 확인.

DB는 미변경이므로 schema 롤백 불요. 신규 이체 행의 use_other_name=0 데이터는 그대로 두어도 무해.
