# 개발 기획서 — 사용자 흐름 UI 정리 (5/19)

## 1. 배경

D-day(2026-05-20)를 하루 앞두고 사용자 화면을 운영자 동선에 맞춰 가다듬는다. 5/19 D-1 리허설 중 다음 5가지가 답답하거나 불필요하다는 피드백이 나왔다.

| # | 페이지 | 현상 | 사용자 요청 |
| --- | --- | --- | --- |
| 1 | CompletePage (`/orders/:id/complete`) | "이체 완료하고 확인 요청" 위·아래 모두 콘텐츠로 빽빽해서 화면이 답답함. 페이지 진입 후 메뉴로 돌아갈 동선이 없음. | 하단 "🍗 조리 현황 보기" 버튼 제거 + 상단에 메뉴로 가는 뒤로가기(back-bar) 추가 |
| 2 | StatusPage (`/orders/:id/status`) ORDERED | "이체 완료 요청" 버튼 위에 단계 진행바(OrderTimeline)가 있어 화면이 납작 | OrderTimeline 제거 |
| 3 | TransferReportForm (TransferPage 내부) | "다른 이름으로 이체했어요" 체크박스가 있으나 운영상 이 경로를 안 씀 | 체크박스 + 조건부 필드 + 서버 매칭 로직 모두 정리 |
| 4 | StatusPage 모든 상태 (ORDERED·TRANSFER_REPORTED·PAID·COOKING·READY 등) | 하단 sticky-bar 안에 "현재 상태 + StatusChip" 라벨 행이 stage-copy 본문 텍스트와 중복 정보 | "현재 상태" 라벨 행 제거 (전 상태 공통) |
| 5 | (#4와 동일) | 확인/조리/수령 단계에서도 동일 노출 | (#4 한 줄 제거로 모든 상태에서 동시 해결) |

## 2. 목표

- 사용자 측 결제·이체·상태 추적 흐름의 시각 잡음을 줄이고, 진입 후 메뉴로 돌아갈 동선을 제공한다.
- 운영자가 안 쓰는 "다른 이름 이체" 경로를 클라이언트·서버 양쪽에서 제거한다.
- D-day 운영 중 사용자가 화면을 한눈에 이해할 수 있게 한다.

## 3. 변경 범위

### 3.1 클라이언트 (사용자 측)

| 영역 | 파일 | 변경 |
| --- | --- | --- |
| CompletePage | `src/pages/customer/CompletePage.jsx` | sticky-bar의 ghost 버튼 1개 제거 + 페이지 최상단에 `<div className="back-bar">` 추가 (뒤로 → `/menu`) |
| StatusPage | `src/pages/customer/StatusPage.jsx` | `<OrderTimeline />` 사용 + import 제거 / sticky-bar 안 "현재 상태 + StatusChip" 라벨 행 제거 + `StatusChip` import 정리 |
| TransferReportForm | `src/components/organisms/TransferReportForm.jsx` | Checkbox + 조건부 otherName Input + 관련 state(`useOtherName`,`otherName`,`showOtherName`)·validation(`errors.otherName`)·제출 페이로드(`useOtherName`,`otherName`) 제거 / 사용 안 하면 Checkbox import 정리 |

### 3.2 서버 측

| 영역 | 파일 | 변경 |
| --- | --- | --- |
| 이체 매칭 도메인 | `server/domain/transfer-matching.js` | `c.use_other_name ? c.other_name : c.depositor_name` 분기 제거 → `depositor_name` 단일화 |
| 고객 API | `server/routes/customer.js` | TransferReport 핸들러의 `useOtherName`/`otherName` 입력 무시 (스키마/검증에서 제거) |
| 관리자 API | `server/routes/admin.js` | `use_other_name`/`other_name` 표시·조회 분기가 있으면 정리 |
| 주문 레포 | `server/repositories/order-repo.js` | 이체 신고 INSERT/UPDATE 시 `use_other_name=0`, `other_name=NULL` 하드코딩 또는 컬럼 미사용 |

### 3.3 DB 스키마 (의도적 비범위)

D-day -1 시점에 운영 DB 마이그레이션은 위험. **컬럼 자체는 유지**하되 신규 데이터에 NULL/0이 쌓이게만 함. ADR-033 사고 재발 방지 차원에서도 schema 변경은 회피.

### 3.4 회귀 테스트 수정

- `server/domain/__tests__/transfer-matching.test.js` — `use_other_name` 케이스 제거 또는 단순화
- `server/repositories/__tests__/order-repo.test.js` — 관련 케이스 정리
- `src/components/organisms/__tests__/TransferReportForm.test.jsx` — 체크박스·조건부 필드 케이스 제거
- 기타 영향 받는 테스트 grep으로 점검

## 4. 비범위 (변경 안 함)

- `src/pages/admin/OrderDetailPage.jsx`의 헤더 StatusChip — 운영자용, 사용자 요구사항 밖.
- StatusPage의 `stage-copy` 본문 텍스트("주문 접수됨", "본부 확인 중", "조리 중!" 등) — 사용자가 "본부 확인 중 보인다"고 명시 → 유지.
- StatusPage의 MascotState · DogTagFrame — 유지 (시각 핵심 컨텐츠).
- DB 컬럼 자체 제거 — 마이그레이션 위험.
- ADR/문서 (CLAUDE.md 등) — UI/로직 정리 후 별도 보강 단계에서.

## 5. 운영 영향

- **D-day 운영자**: "다른 이름 이체" 매칭 화면은 사라지지만, 운영진은 이 경로 사용 안 했으므로 영향 없음.
- **사용자**: 메뉴 → 카트 → 주문 → 입금 안내(CompletePage) → 이체 폼 → 상태 페이지 흐름은 동일. 시각만 더 명료해짐. 메뉴 진입 동선 추가 (CompletePage에 back-bar).
- **운영 데이터**: 신규 이체 신고 행에 `use_other_name=0`, `other_name=NULL`로 저장됨. 기존 행(있다면)은 그대로 보존.

## 6. 가정 (사용자 확인 완료)

- "다른 이름 이체" 서버 로직도 함께 제거 (완전 정리). ✅ 확인됨.
