## 1. CompletePage — 조리 현황 보기 제거 + back-bar 추가

**파일**: `src/pages/customer/CompletePage.jsx`

### 1.1 `<section>` 직후, `<div className="dogtag-stage">` 앞에 back-bar 삽입 (라인 62-63 사이)

```jsx
return (
  <section data-testid="complete-page">
    <div className="back-bar">
      <button type="button" onClick={() => navigate('/menu')} aria-label="뒤로">←</button>
      <h1>주문 완료</h1>
      <span className="meta">#{order.no ?? id}</span>
    </div>
    <div className="dogtag-stage">
    ...
```

- 뒤로가기 경로 = `/menu` (사용자 요청 "메뉴 목록 보이게 하는 뒤로가기").
- 제목 "주문 완료" — TransferPage가 "💸 이체 확인 요청", StatusPage가 "주문 #N" 패턴인 것과 같은 톤.
- 우측 meta `#{order.no ?? id}` — TransferPage 패턴 그대로.

### 1.2 ghost 버튼 제거 (라인 111-113)

before:
```jsx
<div className="sticky-bar" style={{ position: 'fixed', left: 0, right: 0, bottom: 0, zIndex: 40, flexDirection: 'column', gap: 6 }}>
  <button type="button" className="btn btn-primary btn-lg btn-block" onClick={() => navigate(`/orders/${id}/transfer${tokenQuery}`)}>
    이체 완료하고 확인 요청
  </button>
  <button type="button" className="btn btn-ghost btn-block" onClick={() => navigate(`/orders/${id}/status${tokenQuery}`)} aria-label="조리 현황 보기">
    🍗 조리 현황 보기
  </button>
</div>
```

after:
```jsx
<div className="sticky-bar" style={{ position: 'fixed', left: 0, right: 0, bottom: 0, zIndex: 40 }}>
  <button type="button" className="btn btn-primary btn-lg btn-block" onClick={() => navigate(`/orders/${id}/transfer${tokenQuery}`)}>
    이체 완료하고 확인 요청
  </button>
</div>
```

- `flexDirection: 'column'`, `gap: 6` 인라인 스타일 제거 (한 버튼이므로 column 정렬 불필요).

### 1.3 spacer 높이 조정 (라인 106)

```jsx
<div style={{ height: 132 }} />
```
→
```jsx
<div style={{ height: 80 }} />
```

(두 버튼 약 132px → 한 버튼 약 80px)

### 1.4 주석 정리

- 파일 상단 라인 3 "조리 현황 ghost CTA" 언급 부분만 한 단어 정리 가능. Surgical Changes 원칙상 안 건드려도 무해 — 본인 판단.

## 4. 서버 — 이체 매칭 로직 정리

### 4.1 `server/domain/transfer-matching.js`

before (라인 31-35):
```jsx
const candidateName = c.use_other_name
  ? c.other_name?.trim()
  : c.depositor_name?.trim();
```

after:
```jsx
const candidateName = c.depositor_name?.trim();
```

함수 시그니처 JSDoc(라인 14-22)에서 `useOtherName`, `otherName` 언급 없음 → 변경 안 함. 파일 상단 라인 1-9 주석에서 "이름: depositor_name OR (use_other_name=1 시 other_name)" 언급 한 줄 정리 권장.

### 4.2 `server/routes/customer.js`

TransferReport 핸들러 (POST `/orders/:id/transfer-report` 또는 유사 경로)에서:
- request body 스키마(zod)에서 `useOtherName`, `otherName` 제거 또는 ignore
- INSERT/UPDATE 호출 시 `use_other_name=0`, `other_name=null` 하드코딩 (DB NOT NULL 제약 회피)

작업자 행동: 해당 라우트 핸들러를 열고 useOtherName / otherName / use_other_name / other_name 모두 grep → 발견된 부분 정리.

### 4.3 `server/repositories/order-repo.js`

`reportTransfer` 또는 유사 함수에서:
- 매개변수 또는 SQL 바인딩에서 `use_other_name`, `other_name` 사용 제거
- DB schema 컬럼은 유지되므로 INSERT 시 `use_other_name=0`, `other_name=null` 명시 또는 컬럼을 INSERT 컬럼 목록에서 제거 (default 사용)

작업자 행동: `use_other_name|other_name` grep → 모든 사용처 정리. SELECT는 컬럼만 가져오고 사용 안 하면 그대로 둬도 무해 (Surgical Changes).

### 4.4 `server/routes/admin.js`

admin이 use_other_name / other_name 컬럼을 응답에 포함시키거나 표시 분기가 있으면 제거. grep으로 점검.

### 4.5 DB schema (`server/db/init.sql`)

**변경 없음.** 컬럼 자체는 그대로. 이유:
- D-day -1 시점, 운영 DB 마이그레이션은 위험
- 신규 데이터에 NULL/0이 쌓이는 것은 기능 영향 없음
- 향후 정리는 운영 후 별도 작업
