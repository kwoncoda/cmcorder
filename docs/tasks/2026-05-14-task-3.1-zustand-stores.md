# Task 3.1 — Zustand store 설계 (cart · ui · businessState)

## 목표

3 slice 글로벌 상태를 zustand로 도입하고, *셀렉터 강제* 패턴을 회귀로 못박는다.

- `cart`: items 배열 + addItem/removeItem/changeQty/clear. totalQty / totalPrice는 *외부 셀렉터 함수*로 계산 (저장 X).
- `ui`: 모달 스택만 (`modalStack: [{ id, props }]`). 스크롤 위치·미니맵 open 상태 따로 X — 미니맵은 modal id `'minimap'`으로 통합.
- `businessState`: status · operating_date. `shouldBeOpen`은 일정표 + 현재 시각 기반 *셀렉터* (저장 X).
- order slice 미생성 — Complete/Status/Transfer 페이지에서 `useParams().id` 로 fetch (§3.5 3조).
- devtools middleware는 `import.meta.env.DEV` 가드 — production 번들 제외 (§3.5 8조).

## 한 일

### 1. zustand 설치

```
npm i zustand
```

v5.0.13 설치. v5는 selector 결과를 `Object.is`로 비교 (v4의 기본 `===` 와 사실상 동일하므로 회귀 무해). `shallow`는 `zustand/shallow`의 `useShallow` hook을 통해 사용해야 하지만, Task 3.1 셀렉터들은 *primitive 값* (number/boolean/object|null) 반환이므로 shallow 불필요.

### 2. RED — 테스트 3개 먼저

- `cart.store.test.js`, `businessState.store.test.js`, `cart-selector-rerender.test.jsx` 작성 후 `npx vitest run src/store/__tests__` → 3 파일 모두 import resolve 실패 (RED).

### 3. GREEN — store 3개 구현

핵심 결정:

- **cart**: 동일 menuId 중복 추가 시 quantity 누적 병합. `addItem(null)` 또는 `addItem({})` 같은 방어 케이스는 *state 변경 없이 반환* (현재 state 그대로). `changeQty(id, 0)` 또는 음수 시 자동 제거 — UI에서 0 누르기 == 삭제 UX와 일치.
- **ui**: `openModal(id, props)` 시 같은 id가 이미 스택에 있으면 *기존 entry 제거 후 푸시* — 중복 X + 최신 props 갱신. `closeAllModals()` 제공.
- **businessState**: `shouldBeOpen` 셀렉터는 `new Date()` 직접 호출 — 테스트에서 `vi.useFakeTimers() + vi.setSystemTime()`로 결정론적 검증. 일정표는 모듈 상수 `SCHEDULE`로 export — Task 2.9 OPERATING_SCHEDULE 과 동기화 (이후 Phase 4/5에서 단일 출처 통합 가능).

셀렉터 패턴:

```js
// store 정의 (cart.js)
export const cartSelectors = {
  totalQty: (s) => s.items.reduce((sum, i) => sum + i.quantity, 0),
  totalPrice: (s) => s.items.reduce((sum, i) => sum + i.basePrice * i.quantity, 0),
};

// 사용 (페이지/Organism)
const totalQty = useCartStore(cartSelectors.totalQty);   // ✅
const { totalQty } = useCartStore();                     // ❌ (전체 객체 구독 — §3.5 2조 위반)
```

devtools 가드:

```js
const useCartStore = import.meta.env.DEV
  ? create(devtools(cartImpl, { name: 'CartStore' }))
  : create(cartImpl);
```

production 빌드에서 devtools chunk 제외 확인 (grep devtools → 0).

### 4. 셀렉터 회귀 — §3.5 2조 핵심

`cart-selector-rerender.test.jsx`가 보장하는 것:

- `useCartStore(cartSelectors.totalQty)`만 구독한 컴포넌트는, items 배열의 각 항목 객체 reference를 새로 만들어도 *totalQty 합이 동일하면 리렌더되지 않는다*. → §3.5 2조 셀렉터 강제의 실질 가치.
- 안티패턴 (`useCartStore()` 전체 구독)은 동일 시나리오에서 리렌더 발생 → 대조군으로 셀렉터의 효용 입증.

이 회귀는 Phase 4 (MenuPage·StickyCartBar) 구현에서 카트 변동이 메뉴 카드 리렌더를 유발하지 않도록 보장하는 안전장치.
