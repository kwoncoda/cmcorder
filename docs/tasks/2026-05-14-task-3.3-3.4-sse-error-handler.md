### 2. GREEN — useOrderStream

핵심 결정:

- **EventSource jsdom 미구현 → mock 폴리필.** 테스트에서 `class MockEventSource { … }`를 `global.EventSource`에 주입. `onopen`은 비동기 setTimeout 0으로 발화해 실제 네트워크와 유사.
- **인증 분기.** authToken 있으면 URL `?token=<값>` 쿼리 + `withCredentials=false`. 학생(토큰 없음)은 세션 쿠키 사용 + `withCredentials=true`. EventSource는 GET only — POST body 인증이 안 되므로 쿼리 + 쿠키 둘 중 하나.
- **status 전이 감지** — `prevStatusRef`로 직전 status 보관. 첫 snapshot은 `prevStatus === null`이라 콜백 X. 이후 `nextStatus !== prevStatus` 시에만 1회 호출. 동일 status 재전송 시 호출 X.
- **handleSnapshot 공통 함수** — 'message'(전체 교체) / 'status'(부분 merge) 둘 다 status 전이 콜백 발화. merge 옵션으로 분기.
- **business-closed 이벤트** — snapshot에 `business_closed: true` 플래그만 설정. 호출자가 별도 처리 (예: `/closed` redirect). hook은 단순화.
- **error 핸들러** — EventSource는 브라우저 기본 재연결 정책에 위임 (custom backoff 없음 — KISS, ADR-015 의도). `setIsConnected(false)`만, 에러 throw X. 재연결 성공 시 `onopen`이 다시 발화해 `isConnected=true`.
- **latest ref 패턴** — `onStatusChangeRef`를 별도 `useEffect`로 갱신. 메인 effect deps에서 `onStatusChange` 제외 → 호출자가 inline 함수로 매 렌더 새 핸들러 넘겨도 EventSource 재생성 X.
- **cleanup** — `return () => { es.close(); setIsConnected(false); }`. StrictMode mount-unmount-mount 시 첫 EventSource는 close되고 두 번째만 활성 — connection counter ≤ 1.
- **§3.5 5조 준수** — 부수효과(진동·깜박)는 effect 안이 아닌 `onStatusChange` 콜백에서. effect deps에 `status` 없음. 호출자(StatusPage)가 `(prev, next) => navigator.vibrate()` 등으로 처리.

### 3. GREEN — useGlobalErrorHandler

핵심 결정:

- **책임 분리** — 본 hook은 'unhandledrejection'만. 컴포넌트 렌더 에러는 기존 `ErrorBoundary.jsx`. 이벤트 핸들러·비동기 callback의 throw가 누락된 `await` 등으로 promise rejection으로 흘러가면 본 hook이 catch.
- **BusinessClosedError 가드** — `instanceof` 체크. 콜백 있으면 `event.preventDefault()` 후 호출 (콘솔 unhandledrejection 경고 차단). 콜백 없으면 일반 에러 path (console.error) — 안전한 폴백.
- **listener 제거** — `useEffect` cleanup에서 `removeEventListener`. unmount 후 더는 호출 X.
