# Task 3.2 — API client (fetch wrapper + retry + zod + AbortController)

## 목표

`src/api/client.js` + `schemas.js` (zod) + `routes.js` 상수 + `hooks/useApi.js` 도입.
페이지에서 일관된 데이터 fetch 패턴을 강제하고, 다음 회귀들을 못박는다:

- **G13 단일 reactive 진입점** — 어떤 API 호출이든 HTTP 423 응답이면 `BusinessClosedError` throw. `useApi`가 catch하여 `onBusinessClosed` 콜백 발화 → 호출자가 `/closed` redirect. **별도 폴링 X**.
- **5xx 재시도** — 최대 2회, exponential backoff 200ms → 600ms. 4xx·423·AbortError·ValidationError는 재시도 X.
- **zod 런타임 검증** — 응답 본문을 스키마로 검증, 실패 시 `ValidationError`.
- **AbortController 자동 관리** — `useApi`가 useEffect cleanup에서 controller.abort(). StrictMode 2회 mount에도 최종 활성 fetch 1개만 보장.
- **timeout 10s** — 내부 timer가 controller를 abort.

### 3. GREEN — 4 파일 구현

핵심 결정:

- **에러 계층** — `ApiError`가 베이스. `BusinessClosedError`·`ValidationError`가 상속. `instanceof` 또는 `name` 둘 다 회귀 가능. catch 측 분기는 `instanceof BusinessClosedError` 권장 (서브클래스 안정).
- **재시도 로직** — `for (let attempt = 0; attempt <= RETRY_BACKOFFS.length; attempt++)` 패턴. 총 시도 = 1 + 2 = 3.
  - 5xx + 마지막 시도 아님 → `await sleep(backoff)` 후 `continue` (루프 재진입).
  - 5xx + 마지막 시도 → `ApiError` throw.
  - 4xx (400/401/403/404 등) → 즉시 `ApiError` throw, 재시도 X.
  - 423 → 즉시 `BusinessClosedError` throw, 재시도 X.
  - AbortError → 즉시 throw, 재시도 X (취소는 의도된 동작).
  - 네트워크 에러 (fetch reject) → 재시도 가능 (5xx와 동일 경로).
- **AbortController 합성** — 외부 `signal` (호출자 cleanup) + 내부 timeout controller. `AbortSignal.any([...])` 우선 사용, 미지원 시 폴백 (수동 propagation).
- **zod 검증** — `schema.safeParse(data)`. 성공 시 `validated.data` 반환 (z.transform 등으로 데이터가 가공될 수 있어 raw 대신 validated 사용).
- **에러 본문 파서** — 두 가지 백엔드 포맷 지원: `{ ok: false, error: { code, message } }` (API_DRAFT 표준) + `{ error: 'CODE', message }` (단순 포맷). `extractErrorInfo` 함수가 통합.
- **useApi** — `useEffect`가 새 `AbortController` 생성, fetcher에 signal 전달, cleanup에서 abort. `controller.signal.aborted` 가드로 stale setState 방지. `onBusinessClosed`는 `useRef`로 latest 보관 — deps 안정.

### 4. timeout 테스트의 fake timers 활용

```js
vi.useFakeTimers();
global.fetch.mockImplementation((_url, init) => new Promise((_, reject) => {
  init.signal.addEventListener('abort', () => {
    const err = new Error('aborted'); err.name = 'AbortError'; reject(err);
  });
}));
const promise = apiFetch('/api/menus', { timeoutMs: 100 });
const assertion = expect(promise).rejects.toThrow();
await vi.advanceTimersByTimeAsync(150);
await assertion;
```

`vi.advanceTimersByTimeAsync`로 내부 `setTimeout(internalController.abort, 100)`을 트리거. fetch mock이 signal에 반응해 AbortError reject — 결정론적 timeout 검증.

### 5. StrictMode 2회 mount 회귀

`useApi.test.jsx`에서 deps 변경으로 시뮬:

- 첫 mount → fetcher 호출 1 (signal addEventListener 등록).
- `rerender({ deps: [1] })` → 이전 effect cleanup → `controller.abort()` → abortedCount++.
- 새 effect → fetcher 호출 2 → 즉시 resolve.
- 검증: `abortedCount >= 1` (이전 fetch abort 되었음).

실제 StrictMode (React 18 dev)와 동일한 패턴 — useEffect의 mount→cleanup→mount 시퀀스에서 첫 fetch가 abort 되어 setState X.
