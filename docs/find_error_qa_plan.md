# find_error 수정 검수 계획서

## 검수 목적

이번 검수의 목적은 버그 수정 후 학교 축제 현장에서 실제 주문/입금/조리/수령 흐름이 끊기지 않는지 확인하는 것이다.

특히 사용자는 주문 후 현재 상태를 계속 확인할 수 있어야 하고, 관리자는 `이체 확인 요청 → 확인/보류 → 조리중 → 수령대기 → 완료`를 안정적으로 처리할 수 있어야 한다.

## 검수 환경

- 로컬 개발 환경
  - 프론트 Vite 개발 서버
  - Express/SQLite 서버
- 모바일 브라우저
  - iOS Safari
  - Android Chrome
- 관리자 화면
  - `/admin/login`
  - `/admin/dashboard`
  - `/admin/transfers`
  - `/admin/orders/:id`
- 실제 배포 환경 또는 Docker/Nginx 환경
  - `docker-compose.yml`
  - `nginx/default.conf`
  - 정적 SPA fallback과 API reverse proxy 확인

## 테스트 시나리오

### 시나리오 1: 메뉴 이미지/로고 검수

- 홈 추천 메뉴 이미지 확인
- 후라이드/뿌링클 등 recommended 표시 제거 또는 정상 배지화 확인
- 인벤토리/장바구니에서 이모지가 아니라 실제 배그 컨셉 이미지가 보이는지 확인
- `오늘 저녁은 치킨이닭!` 왼쪽 로고가 검은색 단색으로 깨지지 않는지 확인
- 관리자 메뉴 관리에서 이미지가 정상 표시되는지 확인

### 시나리오 2: 주문 생성 후 입금 안내 검수

- 메뉴 담기
- 주문 생성
- 주문번호 발급 확인
- 계좌번호 복사 클릭
- `국민은행 233001-04-403536`만 복사/표시되는지 확인
- 예금주 `박동빈`이 복사 문구에 포함되지 않는지 확인
- 금액 복사 기능이 계좌 복사 수정의 영향을 받지 않는지 확인

### 시나리오 3: 사용자 이체 완료 요청 검수

- 주문 상태 페이지 진입
- 입금 대기중 상태 확인
- `이체 완료 요청` 버튼 노출 확인
- 버튼 클릭 후 이체 완료 요청 화면 또는 모달 진입 확인
- 요청 제출 후 상태가 `이체 확인 요청`으로 변경되는지 확인
- 새로고침 후에도 상태 유지 확인
- 홈으로 갔다가 다시 주문 상태 페이지로 접근 가능한지 확인
- token이 없는 경우 안내가 명확한지 확인

### 시나리오 4: 경과 시간 계산 검수

- 이체 완료 요청 직후 0~1분 수준으로 표시되는지 확인
- 몇 분 후 실제 경과 시간과 일치하는지 확인
- 540분처럼 비정상 표시되지 않는지 확인
- timezone/date parsing/localStorage/server timestamp 문제 확인
- 5분 이상 warning, 10분 이상 danger 강조가 의도대로 동작하는지 확인

### 시나리오 5: 관리자 이체 확인 요청 검수

- 관리자 페이지 진입
- 이체 확인 요청 주문 표시 확인
- 해당 주문 클릭
- `주문을 불러올 수 없어요` 오류가 발생하지 않는지 확인
- 주문 상세가 정상 표시되는지 확인
- design-bundle 기준 `확인`, `보류` 버튼 표시 확인
- 확인 클릭 시 상태 전환 확인
- 보류 클릭 시 보류 처리 확인
- 보류 사유 입력/표시 정책이 있다면 저장 여부 확인

### 시나리오 6: 관리자 주문 상태 칸반/상태 이동 검수

다음 상태별 액션 버튼과 이동 가능 여부를 확인한다.

- 주문 중
- 입금 대기
- 이체 확인 요청
- 이체 완료
- 조리중
- 수령대기
- 보류
- 완료

각 상태마다 확인할 항목:

- 표시 위치
- 가능한 버튼
- 클릭 후 이동 상태
- 실패 시 에러 메시지
- 새로고침 후 유지 여부
- 사용자 status 페이지에 반영되는지 여부

권장 기대값:

| 상태 | 기대 버튼 | 이동 상태 |
|---|---|---|
| 주문 중 / 입금 대기 | 사용자: 이체 완료 요청, 관리자: 취소 | `TRANSFER_REPORTED` 또는 `CANCELED` |
| 이체 확인 요청 | 확인, 보류, 선택적으로 취소 | `PAID`, `HOLD`, `CANCELED` |
| 이체 완료 | 조리 시작 | `COOKING` |
| 조리중 | 조리 완료 또는 수령대기 | `READY` |
| 수령대기 | 완료 또는 전달 완료 | `DONE` |
| 보류 | 재확인, 취소, 사용자 재제출 | 정책 확정 필요 |
| 완료 | 액션 없음 | terminal |

### 시나리오 7: 내역/쿠폰탭 검수

- 현재 비활성화 원인 확인
- 의도적으로 비활성화된 것인지 버그인지 확인
- 활성화해야 한다면 접근 가능 여부 확인
- 숨김 처리한다면 nav에서 혼란이 없는지 확인
- 쿠폰 중복 사용 방지와 내역 표시가 깨지지 않는지 확인
- 개인정보가 관리자에게 필요한 범위 이상 노출되지 않는지 확인

### 시나리오 8: 오픈 예정 문구 검수

- 모든 오픈 예정 문구가 `오후 3시에 오픈할 예정` 기준으로 통일되었는지 확인
- 홈/메뉴/관리자/안내 문구 등 중복 위치 확인
- `/closed` 화면 운영 일정 확인
- 관리자 CLOSED 상태 CTA 안내 확인
- `shouldBeOpen` 계산 기준도 15:00으로 변경해야 하는지 확인

## 자동 테스트 제안

- 주문 상태 전환 단위 테스트
  - `ORDERED -> TRANSFER_REPORTED`
  - `TRANSFER_REPORTED -> PAID`
  - `TRANSFER_REPORTED -> HOLD`
  - `PAID -> COOKING`
  - `COOKING -> READY`
  - `READY -> DONE`
- 이체 완료 요청 API 테스트
  - token 없음 401
  - token 불일치 403
  - 정상 요청 시 `TRANSFER_REPORTED`
  - 중복 요청 정책 확인
- 관리자 확인/보류 액션 테스트
  - 상세 조회 schema 통과
  - 확인/보류/취소 API 호출
  - 보류 사유 저장 정책이 있다면 검증
- 경과 시간 계산 유틸 테스트
  - UTC timestamp + KST 브라우저 환경
  - `YYYY-MM-DD HH:mm:ss` 문자열 파싱 방지
  - 0분, 5분, 10분 경계
- 계좌번호 복사 문자열 테스트
  - 복사 대상이 `국민은행 233001-04-403536`인지 확인
  - 예금주 미포함 확인
- 라우팅 테스트
  - `/orders/:id/status?token=...`
  - `/orders/:id/transfer?token=...`
  - 홈에서 최근 주문 재진입
- 쿠폰 중복 사용 테스트
  - 동일 학번/이름 중복 차단
  - 정산 쿠폰 요약 유지
- 이미지 렌더링 테스트
  - cart item에 `image`가 유지되는지
  - 이미지 없을 때만 fallback 표시

## 수동 QA 체크리스트

- [ ] 현재 브랜치가 `find_error` 또는 수정 작업 브랜치인지 확인했다.
- [ ] 메뉴 화면에서 실제 아이템 이미지가 보인다.
- [ ] 인벤토리에서 실제 아이템 이미지가 보인다.
- [ ] recommended 도장/배지가 사용자 요구대로 정리되었다.
- [ ] 헤더 로고가 검은 단색 블록으로 보이지 않는다.
- [ ] 주문 생성 후 주문번호가 표시된다.
- [ ] 계좌번호 복사 시 `국민은행 233001-04-403536`만 복사된다.
- [ ] `이체 신고` 문구가 사용자 화면에서 남아 있지 않다.
- [ ] status 페이지 `ORDERED` 상태에서 `이체 완료 요청` 버튼이 보인다.
- [ ] 이체 완료 요청 후 `TRANSFER_REPORTED` 상태가 된다.
- [ ] 홈으로 갔다가 status 페이지로 다시 들어갈 수 있다.
- [ ] 새로고침 후 status 페이지가 유지된다.
- [ ] 관리자 이체 확인 요청 카드를 클릭해도 상세 오류가 나지 않는다.
- [ ] 관리자 `확인` 버튼으로 `PAID` 또는 확정된 다음 상태로 이동한다.
- [ ] 관리자 `보류` 버튼으로 `HOLD` 상태가 된다.
- [ ] `PAID -> COOKING -> READY -> DONE` 이동이 가능하다.
- [ ] 경과 시간이 실제 시간과 맞고 540분 오차가 없다.
- [ ] 내역/쿠폰탭 정책이 확정되어 UI에 반영되었다.
- [ ] 오픈 예정 문구가 `오후 3시에 오픈할 예정`으로 통일되었다.
- [ ] 모바일 브라우저에서 sticky 버튼이 겹치거나 잘리지 않는다.
- [ ] Docker/Nginx 배포 환경에서 새로고침 라우팅이 깨지지 않는다.

## 최종 통과 기준

- 사용자가 주문 후 status 페이지를 통해 현재 상태를 계속 확인할 수 있다.
- 사용자가 `이체 완료 요청`을 명확히 제출할 수 있다.
- 관리자가 `이체 확인 요청` 주문을 열고 `확인` 또는 `보류`할 수 있다.
- 관리자가 주문을 `이체 완료 → 조리중 → 수령대기 → 완료`로 이동시킬 수 있다.
- 인벤토리 이미지, 로고, 계좌 복사 문구, 오픈 예정 문구가 사용자 요구와 일치한다.
- 540분 경과 같은 timezone 오차가 재발하지 않는다.
- 내역/쿠폰탭은 활성화, 숨김, 준비중 중 하나로 명확히 정리되어 운영자가 혼란스럽지 않다.
- 자동 테스트와 수동 QA 체크리스트가 모두 통과한다.
