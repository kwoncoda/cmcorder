# find_error 버그 수정 개발 지시서

## 목표

이번 수정의 최종 목표는 학교 축제 주문 운영 중 사용자가 주문 진행 상태를 잃지 않고, 관리자가 이체 확인부터 수령 완료까지 상태를 끊김 없이 처리할 수 있게 만드는 것이다.

핵심 흐름은 다음과 같이 이어져야 한다.

`주문 생성 → 입금 안내 → 이체 완료 요청 → 관리자 확인/보류 → 조리중 → 수령대기 → 완료`

이번 문서는 개발 지시서이며, 이 작업에서는 실제 소스 수정은 하지 않는다.

## 수정 범위

- 메뉴/인벤토리 이미지 렌더링 수정
- 추천 배지/recommended 표시 정책 정리
- 로고 이미지 렌더링 수정
- 계좌번호 복사 문구 수정
- 사용자 주문 상태 페이지/진입 경로 설계
- 이체 완료 요청 버튼 및 페이지/모달 설계
- `이체 신고` 문구를 `이체 완료 요청` 계열 문구로 정리
- 이체 요청 경과 시간 계산 수정
- 관리자 이체 확인 요청 상세 조회 오류 수정
- 관리자 상태 변경 액션 확인/보류/보류해제/조리중/수령대기/완료 처리 설계
- 내역/쿠폰탭 비활성화 원인 확인 및 활성화/숨김 정책 결정
- 오픈 예정 문구 수정

## 핵심 플로우 설계

### 사용자 주문 플로우

1. 사용자가 메뉴를 담는다.
2. 인벤토리/장바구니에서 실제 이미지가 유지된다.
   - 현재 `cart` store는 `image`, `code`, `sub`를 저장하지 않아 이미지가 이모지 fallback으로 떨어진다.
   - 수정 시 주문 API payload는 기존처럼 `menu_id`, `quantity`만 보내 서버 가격 권위를 유지한다.
3. 주문 생성 후 주문번호를 받는다.
   - 현재 화면 표시 주문번호는 `order.no`, 라우트는 DB `order.id`를 사용한다.
4. 입금 대기 상태가 된다.
   - 현재 별도 `DEPOSIT_PENDING` enum은 없고 `ORDERED`가 `주문 접수됨 — 입금 대기 중` 의미를 함께 가진다.
5. 사용자는 주문 상태 페이지에서 계좌번호를 확인한다.
6. 계좌번호 복사 시 `국민은행 233001-04-403536`만 복사/표시된다.
   - 계좌번호 자체는 변경하지 않는다.
   - 예금주 `박동빈`을 화면 안내에 유지할지는 확인 필요.
7. 사용자가 이체 후 `이체 완료 요청` 버튼을 누른다.
   - 완료 페이지뿐 아니라 `ORDERED` 상태의 status 페이지에서도 이 버튼이 보여야 한다.
8. 주문 상태가 `TRANSFER_REPORTED`로 변경된다.
   - 화면 문구는 `이체 확인 요청` 또는 `이체 완료 요청 접수`로 통일한다.
9. 사용자는 status 페이지에서 현재 진행 상태를 계속 확인할 수 있다.
   - token이 URL 또는 storage에 있어야 한다.
   - token이 없을 때 안내 화면이 필요하다.

### 관리자 플로우

1. 관리자 페이지에서 `이체 확인 요청` 주문을 볼 수 있다.
2. 해당 요청을 클릭하면 주문 상세를 정상적으로 불러온다.
   - 현재 `GET /admin/api/orders/:id` raw DB 응답과 프론트 `OrderSchema` 불일치 가능성이 크다.
   - 특히 `is_external`이 SQLite `0/1`인데 schema는 boolean을 기대한다.
3. 관리자에게 design-bundle 기준의 `확인`과 `보류` 액션이 보인다.
   - 현재 칸반 카드에는 액션이 없고 상세 페이지에만 있다.
4. 확인 시 주문 상태가 `PAID` 또는 바로 `COOKING`으로 넘어간다.
   - 권장: 상태 모델 보존을 위해 `PAID`로 먼저 이동하고 이후 `조리 시작`으로 `COOKING` 이동.
   - 바로 `COOKING` 전환은 확인 필요.
5. 보류 시 보류 상태와 사유가 기록된다.
   - 현재 DB에는 `hold_reason` 컬럼이 있으나 관리자 transition API는 reason을 받지 않는다.
6. 이후 `COOKING → READY → DONE` 상태로 이동할 수 있다.
7. 주문 중, 이체 완료, 조리중, 수령대기, 보류 상태별 액션 버튼을 design-bundle 기준으로 정리한다.

### 주문 상태 확인 플로우

사용자가 홈으로 돌아가도 현재 주문 진행 상태를 알 수 있어야 한다.

현재 프로젝트 기준으로는 `/orders/:id/status` 라우트가 이미 존재한다. 다만 `:id`는 화면 주문번호 `no`가 아니라 DB `id`이다.

반드시 다음을 검토한다.

- 주문 완료 후 status 페이지로 자동 이동하는지
  - 현재는 `/orders/:id/complete`로 이동 후 사용자가 버튼을 눌러 status나 transfer로 이동한다.
- 홈 화면에서 최근 주문/주문번호로 status 페이지 재진입이 가능한지
  - 현재 `/menu`에는 최근 주문 재진입 UI가 없다.
- 새로고침 후에도 주문 상태 확인이 가능한지
  - 현재 `useOrderToken`이 URL `?token=` 또는 `sessionStorage`에서 token을 가져오므로 같은 세션에서는 가능하다.
- `localStorage/sessionStorage/server state` 중 무엇을 쓰는지
  - 현재는 `sessionStorage`의 `order:${id}:token`을 사용한다.
  - 브라우저 세션 종료 후에도 재진입이 필요하면 `localStorage` 또는 서버 조회 정책이 필요하다.
- 주문번호만 알아도 조회 가능한지, 보안상 추가 식별값이 필요한지
  - 현재는 주문번호만으로 조회할 수 없다.
  - 주문번호 조회를 허용하려면 token, 학번/이름, 전화번호 등 추가 식별값 정책을 확정해야 한다.

## 상태 모델 정리

현재 코드 기준 주문 상태는 `ORDERED`, `TRANSFER_REPORTED`, `PAID`, `COOKING`, `READY`, `DONE`, `HOLD`, `CANCELED`이다.

| 현재 상태명 | 화면 표시 문구 | 의미 | 사용자에게 보이는가 | 관리자에게 보이는가 | 가능한 다음 상태 | 필요한 액션 버튼 | 수정 필요 여부 |
|---|---|---|---|---|---|---|---|
| `ORDERED` | 사용자: `주문 접수됨 — 입금 대기 중`; 관리자: `주문중` | 주문 생성됨, 입금 대기 | 예 | 예 | `TRANSFER_REPORTED`, `CANCELED` | 사용자: `이체 완료 요청`; 관리자: `취소` | 필요. status 페이지에 요청 버튼 추가 |
| `TRANSFER_REPORTED` | 현재: `이체 신고 완료 — 본부 확인 대기`; 권장: `이체 확인 요청` | 사용자가 이체 완료를 알림, 관리자 확인 대기 | 예 | 예 | `PAID`, `HOLD`, `CANCELED` | 관리자: `확인`, `보류`, 선택적으로 `취소` | 필요. 문구와 관리자 액션 정리 |
| `PAID` | 사용자: `입금 확인 완료 — 조리 대기`; 관리자: `이체완료` | 관리자 입금 확인 완료, 조리 전 | 예 | 예 | `COOKING`, `CANCELED` | 관리자: `조리 시작` | 필요. 칸반 액션 노출 |
| `COOKING` | `조리 중` | 조리 진행 중 | 예 | 예 | `READY`, `CANCELED` | 관리자: `조리 완료` 또는 `수령대기` | 필요. 버튼명 통일 |
| `READY` | `픽업 준비 완료`, `수령대기` | 수령 가능 | 예 | 예 | `DONE`, `CANCELED` | 관리자: `전달 완료` 또는 `완료` | 필요. 사용자 알림/표시 검수 |
| `DONE` | `수령 완료` | 주문 완료 | 예 | 상세/내역에서 필요 | 없음 | 없음 | 확인 필요. 칸반 표시/내역 이동 정책 |
| `HOLD` | `운영진 확인 필요`, `보류` | 입금 정보 불일치 또는 확인 보류 | 예 | 예 | 현재 코드: `PAID`, `CANCELED`; design-bundle: `TRANSFER_REPORTED`, `CANCELED` | 사용자: `이체 정보 다시 보내기`; 관리자: `재확인`, `취소` | 필요. `HOLD -> PAID`와 `HOLD -> TRANSFER_REPORTED` 정책 결정 |
| `CANCELED` | `취소` | 취소됨 | 예 | 상세/내역에서 필요 | 없음 | 없음 | 확인 필요. 취소 사유/표시 정책 |

추가 상태 확인:

- 주문 접수됨: 현재 `ORDERED`
- 입금 대기중: 현재 `ORDERED`의 화면 의미
- 이체 확인 요청: 현재 `TRANSFER_REPORTED`
- 이체 완료: 현재 `PAID`
- 조리중: 현재 `COOKING`
- 수령대기: 현재 `READY`
- 완료: 현재 `DONE`
- 보류: 현재 `HOLD`
- 취소: 현재 `CANCELED`

## Claude 구현 순서 제안

1. 상태 모델/라우팅/주문 조회 구조 파악
   - `src/App.jsx`
   - `src/hooks/useOrderToken.js`
   - `server/routes/customer.js`
   - `server/routes/admin.js`
   - `server/domain/order-state.js`
2. 사용자 주문 상태 페이지와 이체 완료 요청 플로우 수정
   - `ORDERED` status 페이지에서 `이체 완료 요청` 버튼 제공
   - token query 유지
   - 완료 페이지와 status 페이지 문구 통일
3. 관리자 이체 확인 요청 상세 조회 오류 수정
   - 관리자 API serializer 추가 또는 관리자 전용 schema 추가
   - `is_external` 0/1, `amount`, `bank`, `name`, `delivery_type` 등 필드 shape 검증
4. 관리자 상태 변경 액션 정리
   - 칸반 inline 액션과 상세 페이지 액션 중 정책 결정
   - design-bundle 기준 버튼 표시
   - 보류 사유 저장 여부 반영
5. 이미지/로고/recommended/문구 수정
   - cart 이미지 메타데이터 보존 또는 메뉴 데이터 조인
   - recommended 도장 제거/한글화/정책 변경
   - brand logo 표시 방식 검토
   - 계좌 복사 문자열 수정
6. 내역/쿠폰탭 정책 확인 후 수정
   - 숨김, 준비중 페이지, 실제 구현 중 선택
   - 개인정보 노출 범위 검토
7. 시간 계산 버그 수정
   - 서버 timestamp timezone 정책 확정
   - ISO timezone 포함 응답 또는 클라이언트 parser 추가
8. 테스트 추가
   - 상태 전환, 이체 완료 요청, 관리자 확인/보류, 시간 계산, 계좌 복사, cart 이미지, route 재진입
9. 빌드/린트/수동 QA 체크
   - `npm test`
   - `npm run build`
   - 필요 시 `npm run test:e2e`

## 수정 금지/주의 사항

- 계좌번호 자체는 사용자가 준 값 그대로 유지하되, 복사 표시 문구만 수정한다.
- `.env`, 비밀키, 계좌번호 외 민감 설정 파일은 수정하지 않는다.
- 주문/정산 데이터가 깨지지 않도록 마이그레이션 필요 여부를 확인한다.
- 기존 주문 상태와 새 상태가 충돌하지 않도록 주의한다.
- design-bundle과 실제 코드가 다르면 design-bundle을 기준으로 하되, 불명확한 부분은 `확인 필요`로 표시한다.
- 이미지 asset 경로가 없는 경우 임의 이미지를 생성하지 말고 누락으로 기록한다.
- 관리자 `HOLD` 재확인 흐름은 현재 코드와 design-bundle이 다르므로 구현 전에 정책을 확정한다.
- 주문번호 `no`와 DB `id`를 혼동하지 않는다. 사용자 URL 정책을 먼저 정한다.
